# AGENTS.md

Instructions for AI coding agents (Claude Code, Cursor, Codex, etc.) working on this repository. Humans welcome too.

## What this is

A VS Code extension (`danielcregg.vscode-env-ready`) that shows a status-bar banner while waiting for configured extensions to activate, then flips to "ready". Intended for Codespaces / dev containers where the user wants a clear signal that the environment is fully loaded.

Published to the VS Code Marketplace: https://marketplace.visualstudio.com/items?itemName=danielcregg.vscode-env-ready

## Layout

```
src/extension.ts          Entry point. activate()/deactivate(). All runtime logic.
dist/extension.js         Bundled output (esbuild). Shipped in the .vsix. Git-ignored.
out/                      Legacy tsc output. Not used. Delete if you see it.
esbuild.mjs               Build script. --watch for dev, --production for release.
eslint.config.mjs         Flat config, typescript-eslint recommended.
.prettierrc.json          Prettier config (2-space, double quotes, trailing commas, 100 col).
tsconfig.json             Type-check only (noEmit). esbuild does the actual bundling.
.vscode/launch.json       F5 launches an Extension Development Host.
.vscode/tasks.json        watch + check-types tasks with esbuild problem matcher.
.vscode/extensions.json   Recommended extensions for contributors.
.vscodeignore             Files excluded from the .vsix. Keep tight.
.github/workflows/publish.yml  Tag-triggered Marketplace publish (see Release below).
.devcontainer/            Codespaces config. Node 20, github-cli, auto npm install.
images/icon.png           128x128 Marketplace icon.
```

## Scripts

| Command | What it does |
| --- | --- |
| `npm run watch` | esbuild in watch mode. Used by F5 / the default build task. |
| `npm run compile` | One-shot esbuild build (dev mode, sourcemaps on). |
| `npm run check-types` | `tsc --noEmit`. Type-check only. |
| `npm run lint` | ESLint over `src/`. |
| `npm run format` / `format:fix` | Prettier check / write. |
| `npm run package` | `vsce package` → produces `vscode-env-ready-<version>.vsix`. |
| `npm run publish` | `vsce publish`. Runs `vscode:prepublish` (check-types + lint + production esbuild) first. |

## Local development

1. Open in VS Code (or Codespaces — devcontainer auto-runs `npm install`).
2. Press **F5** to launch an Extension Development Host with the extension loaded.
3. Edit `src/extension.ts`. The watch task rebuilds on save; reload the dev host window (Cmd/Ctrl+R) to pick up changes.
4. Configure `envReady.waitFor` in the dev host's settings to test the banner against real extension IDs.

## Release (publish to Marketplace)

The publish pipeline is **tag-triggered**. Regular commits to `main` do not publish.

```bash
# from main, after merging your changes
npm version patch          # bumps package.json version + creates a v<version> git tag
git push --follow-tags     # pushes the bump commit AND the tag
```

The tag push fires `.github/workflows/publish.yml`, which:
1. Verifies the tag matches `package.json` version (catches forgotten `npm version` bumps).
2. Runs `check-types` + `lint`.
3. Runs `vsce publish` using the `VSCE_PAT` secret.

Watch the run at https://github.com/danielcregg/vscode-env-ready/actions. End-to-end ~60-90s.

Use `npm version minor` for features, `npm version major` for breaking changes. Stick to semver.

## Secrets

- **`VSCE_PAT`** (GitHub repo secret) — Azure DevOps Personal Access Token with scope **Marketplace → Manage** and organization **All accessible organizations**. Rotates annually (DevOps max).
  - Recreate at https://dev.azure.com/<org>/_usersSettings/tokens.
  - Update at https://github.com/danielcregg/vscode-env-ready/settings/secrets/actions.

If `vsce publish` returns 401, the PAT has almost certainly expired or was created scoped to a single organization instead of all accessible orgs.

## Conventions an agent should follow

- **Do not commit `dist/` or `out/`.** Both are git-ignored. `dist/` is regenerated by esbuild; `out/` is dead.
- **Do not edit `.vsix` files in-tree.** They're build output. The packaged `.vsix` is git-ignored.
- **Keep `.vscodeignore` tight.** Anything not strictly needed at runtime (dev configs, build scripts, source maps in production, `out/`) should be excluded. The shipped `.vsix` should be ~8 KB / 8 files for this extension. If it grows past ~50 KB you've leaked something.
- **`publisher` in `package.json` is `danielcregg`.** Do not change it — it must match the Marketplace publisher ID, and changing it abandons the existing listing.
- **`main` field points to `dist/extension.js`.** If you switch bundlers or output paths, update this too or the published extension won't load.
- **Bump the version before tagging.** The workflow refuses to publish if tag and `package.json` disagree.
- **Don't add new top-level config files casually.** Prefer extending an existing one (eslint, prettier, tsconfig) over introducing another.

## Common tasks

| Task | How |
| --- | --- |
| Add a new setting | Add it under `contributes.configuration.properties` in `package.json`, read it in `src/extension.ts` via `workspace.getConfiguration("envReady")`. Update README if user-facing. |
| Add a dependency | `npm install <pkg>` for runtime, `npm install -D <pkg>` for dev. Runtime deps get bundled by esbuild — be mindful of `.vsix` size. |
| Change Node version | Update `.nvmrc`, `engines.node` in `package.json`, and `target` in `esbuild.mjs` together. CI reads `.nvmrc`. |
| Investigate a publish failure | Check the Actions run logs first. Tag-vs-version mismatch and expired PAT account for ~95% of failures. |
| Re-test the packaged extension locally | `npm run package`, then in VS Code: **Extensions → ... menu → Install from VSIX**, pick the generated file. |

## Things deliberately not in this repo

- **No test suite yet.** If you add one, wire it into the workflow before the publish step and update this file.
- **No CHANGELOG automation.** `CHANGELOG.md` is hand-edited; update it as part of the version-bump commit, not after.
- **No pre-release channel.** Single `latest` track. If we ever want `--pre-release` builds, that's a workflow addition with a separate trigger.
